# API 设计规范

## 一、RESTful 约定

- 资源使用名词复数，如 `/users`、`/orders`。
- 用 HTTP 方法表达操作：GET 查询、POST 创建、PUT 全量更新、PATCH 部分更新、DELETE 删除。
- 禁止在 URL 中使用动词，如 `/getUser` 应改为 `GET /users/{id}`。

## 二、版本管理

- 在 URL 中带版本号，如 `/api/v1/users`。
- 不兼容变更须升大版本号；向后兼容的新增可在原版本上进行。
- 旧版本至少保留 6 个月过渡期，下线前须提前通知调用方。

## 三、状态码

| 场景 | 状态码 |
| ---- | ------ |
| 成功 | 200 / 201 |
| 参数错误 | 400 |
| 未认证 | 401 |
| 无权限 | 403 |
| 资源不存在 | 404 |
| 服务端错误 | 500 |

## 四、响应格式

统一返回 JSON，结构如下：

```json
{ "code": 0, "message": "ok", "data": {} }
```

- `code` 为业务状态码，0 表示成功。
- 分页接口须返回 `total`、`page`、`page_size`。

## 五、安全与限流

- 所有写接口须鉴权；敏感接口须校验权限。
- 对外接口须限流，默认单 IP 每秒不超过 20 次。
- 禁止在响应中返回密码、密钥等敏感字段。
