# 代码评审规范

## 一、适用范围

本规范适用于公司所有进入主干分支（main / release/*）的代码变更。
私有分支自测代码不强制走评审，但任何对外可见的 PR 必须遵守本规范。

## 二、强制规则

### 2.1 评审人数
- 普通业务代码：至少 1 名同模块同事 +1 名负责人 Review。
- 涉及鉴权、支付、数据迁移：至少 1 名安全 / DBA 同学额外 Review。
- 评审人不得为 PR 作者本人或直接汇报关系的下属。

### 2.2 必过门禁
合并 PR 之前必须满足：
1. CI 全绿（lint、单元测试、集成测试）。
2. 单元测试覆盖率不得低于 70%，新增代码覆盖率不得低于 80%。
3. 至少一位 Reviewer 显式 Approve；驳回（Request Changes）必须解决后才能合并。
4. PR 描述包含：变更动机、影响范围、回滚方案。

### 2.3 禁止事项
- 禁止在评审中"先合后议"——任何未解决的 Comment 必须显式标记 resolved。
- 禁止直接 push 到 main 分支（受 GitHub branch protection 保护）。
- 禁止在 PR 中夹带与主题无关的大规模 refactor，须另开 PR。
- 禁止提交包含 API Key、密码、内网域名的代码（pre-commit hook 拦截）。

## 三、评审重点

按下列优先级检查：

1. **正确性**：业务逻辑是否符合 PRD；边界条件是否覆盖。
2. **安全**：是否存在 SQL 注入、XSS、越权访问、未脱敏日志。
3. **性能**：循环复杂度、N+1 查询、未加索引的数据库访问。
4. **可读性**：命名是否清晰、函数是否过长（建议单函数 ≤ 50 行）。
5. **可维护性**：是否有合理的注释、是否避免重复造轮子。

## 四、SLA

- 工作日提交的 PR，Reviewer 应在 **1 个工作日内**给出首次反馈。
- 周五下午之后提交的 PR，SLA 顺延至下周一。
- 紧急修复（hotfix）走 fast-track 通道，可由值班负责人单独 Approve，但必须 24 小时内补充事后评审。

## 五、争议处理

评审中出现技术分歧时：
1. 先在 PR 中文字讨论；
2. 24 小时内未达成一致的，召集相关方线下会议；
3. 仍有争议的提交技术委员会决议。
