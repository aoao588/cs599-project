# 编码风格规范

## 一、通用原则

代码首先是写给人读的，其次才是给机器执行的。所有代码须遵循"清晰胜于聪明"的原则，
避免炫技式的晦涩写法。统一风格由 lint 工具在 CI 阶段强制检查，不符合规范的代码无法合并。

## 二、命名规范

- 变量、函数使用小驼峰（camelCase）；类名使用大驼峰（PascalCase）；常量全大写下划线分隔。
- 名称须能自解释，避免 `a`、`tmp`、`data2` 等无意义命名。
- 布尔变量以 `is`、`has`、`can` 开头，例如 `isValid`、`hasPermission`。
- 禁止使用拼音或拼音缩写命名。

## 三、函数与文件

- 单个函数建议不超过 50 行，圈复杂度不超过 10。
- 单个源文件建议不超过 800 行，超过应拆分模块。
- 函数参数不超过 5 个，过多应封装为对象。

## 四、注释

- 公共 API 必须有文档注释，说明用途、参数、返回值与异常。
- 注释解释"为什么"而非"做什么"；代码本身应能说明在做什么。
- 禁止保留大段被注释掉的废弃代码，应直接删除（版本管理可追溯）。

## 五、格式化

- 统一使用项目根目录的格式化配置（Prettier / Black / google-java-format）。
- 缩进 4 空格，禁止使用 Tab 与空格混用。
- 提交前须运行格式化，CI 会校验格式一致性。
